# 代码可行性分析（再次遍历后）

## 1. 总体结论
当前仓库**可行**，能够完整跑通：
- 客户端采集并判断活动类型；
- 服务端接收并持久化；
- 页面展示最近日志。

适合作为课程设计/演示原型；若要进入长期、真实监督场景，仍需补充准确率与隐私合规能力。

---

## 2. 再次遍历后的关键事实

### 2.1 服务端（`server.py`）
- 已实现 `/check_activity` 接口，支持 `activity/source/confidence` 入参。
- 已落库 SQLite（`activity_log`），并非内存临时列表。
- 首页 `/` 渲染最近 200 条活动日志。

### 2.2 客户端（`client.py`）
- 主模式：截图 + OCR 文本 + 关键词命中统计。
- 降级模式：当 OCR 不可用时，使用 `psutil` 的进程名规则匹配。
- 上报失败时可本地兜底提示，不会直接崩溃。

### 2.3 前端展示（`templates/index.html`）
- 简单表格页，展示时间、活动类型、消息、来源、置信度。
- 适合原型验证，不涉及复杂前端依赖。

---

## 3. 可行性判断（按目标划分）

### 3.1 原型/课程设计：可行
原因：
- 架构简单、依赖明确、启动成本低；
- 具备降级路径，鲁棒性比单纯 OCR 更好；
- 服务端已有持久化，演示可追溯。

### 3.2 生产/严肃监督：暂不建议直接上线
主要限制：
- 判定逻辑以关键词规则为主，语义理解能力弱；
- 浏览器场景学习/娱乐混用严重，误报与漏报难避免；
- 尚缺访问控制、审计、脱敏、数据保留策略等合规能力。

---

## 4. 运行方式（简要）
1. 安装依赖：`pip install -r requirements.txt`
2. 启动服务端：`python server.py`
3. 启动客户端：`python client.py`
4. 查看日志：打开 `http://localhost:5000/`

可选：通过 `TESSERACT_CMD` 指定 OCR 可执行文件路径；通过 `ACTIVITY_DB_PATH` 指定数据库文件路径。
